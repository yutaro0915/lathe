# explain-diff — 変更の説明教材（自己完結 HTML）を生成する

出自: https://www.geoffreylitt.com/2026/07/02/understanding-is-the-new-bottleneck.html の
/explain-diff を lathe 規約へ適応（PdM 承認 2026-07-05。v1〜v3 の失敗＝要約・演出・前提知識の
仮定、を経て確定）。**要約（圧縮）でなく教材（展開）を作る** — 読者がゼロから世界を
組み立てられること。

## 起動条件

- PdM の要求時（既定）。監査役の判断で governance 級・大型の変更にも推奨。
- **適用範囲追記（PdM 指示 2026-07-06）**: diff 教材に限らず、**PdM 向け報告 HTML 全般**
  （overview・監査報告等）も本 skill の形式・視覚言語（紙面調 CSS・HTML/CSS 図・callout・
  である調・禁則）に統一する。他フォーマット（mermaid ダーク調等）は使わない。
- merge への自動組み込みはしない（simplicity。必要が実証されたら再考）。

## 生成指示（subagent へこのまま渡す・対象と接地先だけ差し替える）

対象変更の rich で interactive な説明 HTML を 1 枚作る。セクションは次の 4 つ・この順。

1. **Background**: 変更に関係する既存システムの説明（周辺コードを広く探索してから書く）。
   読者の前提知識を仮定しない。まず初心者向けの深い背景を「既知なら読み飛ばせる」形で置き、
   続いて変更に直接関わる狭い背景。**登場するすべての主体について「何をするのか・
   なんのために存在するのか」を必ず書く。**
2. **Intuition**: 変更の核心の直感。詳細でなく本質。**toy データの具体例**（架空だが実形式の
   ID・JSON・sha を使った before/after）。**図をふんだんに**。
3. **Code**: 高水準ウォークスルー。ファイル順でなく理解できる順にグループ化し、
   各グループに diff の要点を `<pre>` で引用。
4. **Quiz**: 理解を試す**中難度の 5 問**（実質を理解していないと解けないが、ひっかけでない）。
   クリックで正誤判定と解説が出る interactive 多肢選択（JS 内蔵）。

### 形式

- **単一の自己完結 HTML**（CSS/JS 内蔵・外部依存なし）。1 本の長いページ＋セクション見出し＋
  冒頭に目次。タブ構造は使わない。スマホでも読める簡単な responsive。
- 出力先: `.lathe/reports/YYYY-MM-DD-explain-<slug>.html`（日付 prefix・gitignored）。
- **図は ASCII 禁止・HTML/CSS で構築**。少数の図ファミリーを繰り返し使う
  （データフロー図には**実データ例**を載せる／UI 変更には UI 簡略図）。
- コードブロックは必ず `<pre>`（div なら `white-space: pre-wrap` 必須）。
  **保存前に全コードブロックの CSS を検査**する。
- 重要概念の定義・エッジケースは callout。
- 日本語・である調。Martin Kleppmann 級の明晰さと流れ（節間の遷移を滑らかに）。

### 禁則（v1〜v3 の失敗から。違反は不合格）

- **擬人化・演出・物語調の禁止**（「門番を解雇」等の修辞は書かない。engaging さは
  具体例と積み上げで出す）。
- **相対時間の禁止**（「昨日」でなく日付・PR/ADR 番号の絶対参照。書いた瞬間から腐る語を使わない）。
- **評価ラベルの見出し禁止**（「勘所」「ポイント」等。重要さは構成から立ち上がらせる。
  読者の判断を先取りしない）。
- **接地必須**: 事実は diff・ADR・コード・API 実測から。推測で書かない。不明は「未確認」と明記。

### 重複の扱い（PdM 2026-07-05）

同じ背景解説が複数の教材で繰り返し出てきたら、**その時に**そのセクションだけ正本
（`design/` 等）へ抽出して参照に切り替える。事前の共通化はしない。

## 検収（監査役が生成物に対して機械照合してから PdM へ）

- 禁則語 grep（昨日/門番/運命/勘所 等）・4 節の存在・quiz JS の存在・`<pre>` の white-space。
- 初出の実物例: `.lathe/reports/2026-07-05-explain-pr110-receipt-to-ci.html`（PR #110）。
