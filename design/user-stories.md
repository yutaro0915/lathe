# Phase 1〜6 ユーザーストーリー（仮）

## 目的と使い方

[ROADMAP.md](../ROADMAP.md) の完了条件は「システムが何をできるか」で書かれている。
本ノートは同じ Phase を「dogfood の Yutaro が何をして何を得るか」の形で書き直し、
両者を突き合わせて **ズレ（完了条件が拾えていないストーリー / ストーリーに対応しない完了条件）**
を洗い出す。ズレは G 番号で採番し、末尾で #10（archive v2 / 界面契約）への入力に変換する。

正本はあくまで ROADMAP。本ノートはレビュー後、ROADMAP 改訂 or ADR の入力にして役目を終える。

## 前提（persona）

- Yutaro 1 人。複数 repo（LLMWiki hub / zenn / lathe 自身 等）で Claude Code と Codex を日常運用。
- ハーネス＝ CLAUDE.md / AGENTS.md / skills / hooks / memory。これらを定量的に改善したい。
- 形式: 「**…したとき、…が起き、その結果 …する**」。結果節は必ず Yutaro の行動で終える
  （システムの出力で終えない。出力で終わるストーリーは価値が未検証）。

---

## Phase 1 — 観測

- **S1-1（事後検死）**: セッションが終わったとき、Lathe を開くとそのセッションの全行動
  （メッセージ / ツール / 差分 / thinking / サブエージェント / コスト）が見え、その結果
  transcript の生 JSONL を読まずに「何が起きたか」を 5 分で把握する。
- **S1-2（差分の帰属）**: 想定外の変更ファイルを見つけたとき、diff から transcript の該当
  step へ遡れ、その結果「誰が・どの指示の流れで・なぜ」変えたかを特定する。
- **S1-3（コスト監視）**: 週の終わりに Overview を見たとき、project 横断のコスト・トークン
  消費が見え、その結果異常に高いセッションを開いて原因（失敗ループ等）を確認する。
- **S1-4（自動取り込み）**: 別 repo でセッションを終えたとき、手動 `pnpm ingest` なしで
  Lathe に反映され、その結果「取り込み忘れで観測に穴が空く」ことを心配しなくなる。
- **S1-5（PR 起点の閲覧）**: ある PR を見直すとき、その PR を生んだセッション群が PR に
  紐付いて見え、その結果「意図 → 実装 → review → merge」を 1 本の流れとして読む。

### ROADMAP 完了条件との突き合わせ

| ストーリー | 対応する完了条件 | 判定 |
|---|---|---|
| S1-1, S1-2, S1-3 | 実装済みチェックボックス群 | 整合 |
| S1-4 | 「増分 ingest / watch」 | 整合（#2〜#5 の設計がそのまま実装経路） |
| S1-5 | 「PR 連携（PR 履歴を実行記録に取り込み）」 | **G1: 不足** |

- **G1**: 完了条件は「取り込む」までで、S1-5 の成立には **PR ⇄ セッションの紐付け**が要る。
  紐付けキー（branch / commit SHA / 時間窓 / cwd）は未設計。これは DB スキーマ（論点 #6）と
  archive 設計（#10）に跨る界面契約。「取り込んだが紐付かない」では story が成立しない。

---

## Phase 2 — AI 分析

- **S2-1（finding → ハーネス編集）**: 直近のセッション群を analyst に読ませたとき、
  finding（失敗ループ / 未帰属差分 / 過剰トークン / リスク行動）が根拠リンク付きで出て、
  根拠イベントへジャンプして文脈を確認し、その結果 **CLAUDE.md 等のどこを直すかを決める**。
- **S2-2（再発パターン）**: 横断分析を回したとき、同じハーネスで同じ失敗が複数セッションで
  再発していることが分かり、その結果単発の事故ではなく構造問題として扱い、修正の優先度を上げる。
- **S2-3（finding の棄却）**: finding が的外れだったとき、棄却理由を付けて archive に残し、
  その結果 analyst の精度推移を後から評価できるようにする。

### 突き合わせ

| ストーリー | 対応する完了条件 | 判定 |
|---|---|---|
| S2-1 | 「finding が 1 件でも有意義に出る」「クリックでジャンプ」 | **G2: 「有意義」が無定義** |
| S2-2 | 「複数セッション横断の finding 抽出」（スコープ節） | ほぼ整合 |
| S2-3 | なし | **G3: 不足** |

- **G2**: 「有意義」をストーリー基準で定義すべき。提案: **「finding を読んだ結果、実際に
  ハーネスを 1 箇所編集する意思決定に至ったら有意義」**（actionable 性）。これを Phase 2 の
  完了判定に昇格させると、analyst の出力設計（要約の粒度・根拠の深さ）が逆算できる。
- **G3**: 採否（棄却含む）の記録は ROADMAP では Phase 6「採否管理」まで出てこないが、
  finding の採否は Phase 2 の運用初日から発生する。最小の dec 記録（adopt / reject + 一言）は
  Phase 2 のデータモデルに含めるべき。→ #10 への入力。

---

## Phase 3 — 対照実験

- **S3-1（finding → fixture 化）**: 構造問題と判断した finding があるとき、その finding の
  元になったセッション入力を fixture として固定でき、その結果「あの失敗をもう一度起こせる」
  状態を手に入れる。
- **S3-2（並走比較）**: CLAUDE.md を 1 行変えた改善案があるとき、改善前後のハーネスで同じ
  fixture を並走実行でき、その結果両者の挙動差分（成否・コスト・行動列）を並べて読む。

### 突き合わせ

| ストーリー | 対応する完了条件 | 判定 |
|---|---|---|
| S3-1 | Phase 2 完了条件の「finding は fixture として再利用できる」 | **G4: 操作が未定義** |
| S3-2 | 「改善前 vs 改善後で同じ fixture を回せる」「結果が DB 記録・finding と紐付く」 | **G5: 「読む」側が無い** |

- **G4**: 「再利用できる」の操作 UX が未定義（finding からワンアクションで fixture 化か、
  手で transcript を切り出すのか）。手作業が重いと dogfood でループが回らなくなる急所。
  同時に **fixture が保持すべき入力の範囲**（プロンプトだけか、repo 状態・ハーネス版数込みか）
  は界面契約そのもの。→ #10 への入力（最重要）。
- **G5**: 並走「結果を記録する」は完了条件にあるが「結果を並べて見る UI」が無い。採点（Phase 4）
  前に人間が差分を読める画面が無いと、rubric 設計（Phase 4）の材料が取れない。最小で良いので
  Phase 3 の完了条件に「並走結果の比較ビュー」を足すべき。

---

## Phase 4 — Evals / 自動採点

- **S4-1（rubric の用意）**: 並走結果を採点したいとき、rubric をテンプレートから選んで
  数分で微調整でき、その結果 rubric 作成が実験のボトルネックにならない。
- **S4-2（採否の意思決定）**: judge の採点が出たとき、勝敗と採点理由（根拠リンク付き）を読み、
  その結果改善案を採用するか棄却するかを決める。
- **S4-3（judge への不信）**: 採点が直感に反したとき、採点理由から該当の並走結果・finding へ
  遡って検証でき、その結果 judge 自体の信頼度を自分で較正する。

### 突き合わせ

| ストーリー | 対応する完了条件 | 判定 |
|---|---|---|
| S4-2 | 「機械的に勝敗が出る」「採点理由が finding と紐付く」 | 整合 |
| S4-1 | 「rubric の設計」（スコープ節） | **G6: 運用コストの観点が無い** |
| S4-3 | なし | ほぼ G5 と同根（遡れる記録があれば成立） |

- **G6**: rubric を毎回ゼロから書く前提だと dogfood で回らない。完了条件に「rubric の再利用
  （テンプレ + 上書き）」を含めるか検討。重くするなら Phase 6 送りでも良いが、判断を明示すべき。

---

## Phase 5 — Agent 接続

- **S5-1（改善の適用）**: 採用を決めた CLAUDE.md 改訂案があるとき、`apply()` で対象 repo に
  PR が立ち、その結果 GitHub 上でレビューして merge する（直接書き換えはされない）。
- **S5-2（多 agent 横断）**: 同じハーネス改善を Claude Code と Codex の両方に適用したいとき、
  同じ操作で両者に PR が立ち、その結果 agent ごとの個別作業をしない。

### 突き合わせ

S5-1 / S5-2 は完了条件（apply → PR 化、Codex / Cursor skeleton）とほぼ整合。
「PR 経由・人間レビュー必須」が安全境界としてストーリー側からも要請される点を明文化したのみ。

---

## Phase 6 — 統合

- **S6-1（ループ運用）**: 週に一度ダッシュボードを開いたとき、観測 → 分析 → 改善案 → 実験 →
  採点 → 採用待ちの各段に何が滞留しているかが見え、その結果滞留を 1 クリック範囲で前へ進める。
- **S6-2（品質の推移）**: 改善履歴を時系列で見たとき、ハーネス版数ごとのスコア推移が読め、
  その結果「この 1 ヶ月でハーネスは良くなったか」に数字で答える。
- **S6-3（回帰検知）**: 過去に採用した改善が後のセッションで悪化を示したとき、回帰アラートが
  出て、その結果当該改善を巻き戻すか再実験するかを決める。

### 突き合わせ

| ストーリー | 対応する完了条件 | 判定 |
|---|---|---|
| S6-1, S6-2 | 「1 クリック範囲で回る」「推移が読める」 | 整合 |
| S6-3 | 「回帰アラート」（スコープ節にあり、完了条件には無し） | **G7: 完了条件へ昇格を検討** |

- **G7**: 回帰検知は「採用した改善のスコアを以後のセッションでも追い続ける」ことを意味し、
  archive のスコア記録が**ハーネス版数に紐付いている**ことを要求する。これも #10 への入力。

---

## 横断所見 — #10（archive v2 / 界面契約）への入力

ストーリーから逆算すると、Phase 1 のデータモデルが Phase 2〜6 に対して負う契約は最低限:

1. **イベントへの安定参照**（G2, S2-1, S4-3）: finding・採点理由が張る根拠リンクは、
   再 ingest しても壊れない event ID を要求する。
2. **PR ⇄ セッションの紐付けキー**(G1): #6 の DB スキーマ設計と同時に決める。
3. **finding の採否記録**（G3）: Phase 6 の採否管理を待たず、最小の dec を Phase 2 から。
4. **fixture が固定すべき入力の範囲**(G4): プロンプト列 + repo 状態参照 + **ハーネス版数**。
   ここが archive v2 踏襲度判定の核心。
5. **ハーネス版数の第一級化**（G4, G7): 「どの CLAUDE.md の版で走ったか」が session・
   並走結果・スコアの全てに紐付くこと。回帰検知（S6-3）はこれが無いと成立しない。

## 出口

- 本ノートをレビュー → G1〜G7 の採否を決める → 採るものは ROADMAP の完了条件改訂 +
  #10 検討（lathe-phase7 の archive format v2 spec との突き合わせ）の入力にする。
- 反映後、本ノートは ROADMAP / ADR に吸収して archive する。
