## A. 「小さなカーネル」が 7,000 行に戻る力学

### A-1. 見積り 2.5–3.5k は「数える境界」の産物
- **位置**: §4.7 表・§1-8
- **何が**: カーネル計 2.5–3.5k 行の集計に入っていないものが多い——contracts 群（9 loop × trigger 述語/injection/allowlist/envelope schema・§4.0-1）・Actions yml・ops/ systemd unit 群（dispatcher timer・watchdog timer・proxy socket unit・transient run unit template）・install self-check（M8）・contract test（M12）・DDL/migration・そして「テスト同等以上」＝現行 6,183 行超のテスト。
- **なぜ**: 現行系も「*.mjs 実装だけ」を数えれば小さく見える。実測では現行 scripts/ 非テスト実装は **7,015 行**（2026-07-08 計測。本案の「~6.2k」は preflight 等 drop 分を母数から外した数字）。本案は自分に有利な境界で新旧を数え分けており、「カーネルは小さい」は会計方針の帰結であって構造の性質ではない。

### A-2. 成長率の実測: 現行系は「最小」から 32 日で 7k 行に達した
- **位置**: §1-1「保証の最小核 5 部品だけ」・§4.7
- **何が**: scripts/ の初コミットは **2026-06-06**、今日（07-08）までの **32 日間・98 commits** で非テスト 7,015 行＋テスト 6,183 行に成長した（機械計測）。inner-loop.mjs は 07-02、orchestrator.mjs は 07-07 追加＝**code red 前日まで部品が増え続けていた**。
- **なぜ**: 現行系も同じ監査役・同じ規律（PR+CI・ADR・rubric 48 本・「プロンプトより機械的保証」）の下で建った。本案には「今回は増えない」を保証する**新しい機構が一つもない**（M11 相当は現行にも ADR 0036 として存在した）。事故→対処部品の追加（watchdog 検査 9 項目・post-check 5 項目・補償・contract test…はすべて今回の事故台帳から生えた部品）という増殖動力は自作継続でそのまま残る。次の 26 件の incident 台帳が次の 3k 行を要求する。routines 案では発火・環境・版管理の対処部品が platform 側に生えるため、少なくともその軸の増殖は自 repo に積もらない。

### A-3. driver 800–1,200 行の縮小根拠が自壊している
- **位置**: §4.3
- **何が**: 縮小根拠は「prompts 417＋projects 263 の消滅＋envelope 化の分岐削減」だが、同じ節が「rework（S2-4）と land の gh 癖対処（Q1〜Q3・A3）は複雑さの本体でありなくならない」と自認する。現行 driver 3,387 行の大半はまさにその複雑さの堆積。
- **なぜ**: 散文 prompt→contracts JSON は行数の**移動**であって消滅ではない（inner-loop-prompts.mjs 417 行も当時は「ただのテンプレ」だった）。3,387→800–1,200 は 65〜76% 削減の宣言だが、削減の 7 割を占めるべき「分岐削減」は未実装・未実測。楽観係数が最も大きい数字が最も大きい部品に置かれている。

### A-4. 「loop 定義 = データ」は S2-9 の JSON 版を新設する
- **位置**: §4.0-1（routines 案 §4.1/4.2/4.4 を共通正本として参照）
- **何が**: trigger 述語は評価器（＝コード）を要する事実上の DSL。9 loop 分の述語・注入必須リスト・allowlist は設計判断を JSON に埋め込む。
- **なぜ**: JSON はテスト網が散文より強いとは限らない——述語の論理誤り・allowlist の 1 項目欠落（deny デフォルトなので**欠落 = run 途中の silent tool 拒否**）は、S2-6（settings 暗黙 load）・S2-9（書式クラッシュ）と同型の「契約データの事故クラス」として立つ。本案はこのクラスの検証装置（述語の網羅テスト・allowlist の実行時突合）を部品表に持たない。

## B. 常駐運用負荷の過小評価

### B-1. M8 の 1 行に圧縮された負担は「拡大」する
- **位置**: §3-M8・§1-4
- **何が**: 「routines 案で platform が消した分がここに全部残る」と正直に書くが、残るだけでなく**増える**: 新設計は systemd unit を現行より増やし（timer×2・proxy socket・transient run）、しかも全部**これから新規に書く**。
- **なぜ**: E1（cgroup 回収）は unit 設定（KillMode）の事故だった。unit を書き直す＝同クラスの設定事故をもう一度踏み直す位置に自分を置く。E1〜E3 は 1 台の case で数日内に 3 種踏んだ実績があり、「ops/ 正本化＋self-check で解決」の self-check 自体が未着手のコード（S2-10 は self-check が書かれるまで現役）。

### B-2. Postgres が M1 と M2 の両方の単一依存点になり、その運用が無計上
- **位置**: §4.1 claims DDL・§6-2
- **何が**: 排他（M1）も検知（M2 heartbeat）も同一 DB に依存。DB 不達 = fail-closed 全停止は設計通りだが、**その停止を報じるのは未実測の Actions cron だけ**（§7-7 自認）。backup・vacuum・upgrade・disk・（cross-machine 時）ネットワーク到達と認証が恒久運用に乗る。
- **なぜ**: §6-2 は「置き場（同居 or 専用）」の裁定だけを立て、**DB 運用主体の工数を一度も数えていない**。lathe Postgres 同居なら、dev 作業（scratch DB・コンテナ再起動）と統治系の本番状態が同一インスタンスに同居する新しい結合を作る。routines 案は managed Postgres でこの面を外出ししている。

### B-3. 相関故障: 検知系が被検知系と同じ床に立っている
- **位置**: §4.6
- **何が**: dispatcher・watchdog・heartbeat 書込（proxy 経由）は同一マシン・同一 systemd・同一 DB。E1 級・電源・disk full の 1 発で**監視する側とされる側が同時に死ぬ**。
- **なぜ**: 残るのは GitHub Actions cron のみ。公称 best-effort・混雑時数十分遅延（本案 §7-7 自認・未実測）。加えて public repo の scheduled workflow は**リポジトリ無活動 60 日で自動無効化される仕様**（GitHub docs 記載・本 repo での照合は未実施＝未確認）——マシン死で commit が止まった後、最後の砦自体が silent に止まりうる構造。routines 案基線は発火面が platform に外出しされ、この相関が切れている。

## C. 保守 bus factor（監査役 1 人）

### C-1. セキュリティ境界コードを 1 人で書き 1 人で監査する
- **位置**: §4.5 proxy・§3-M3
- **何が**: 系で唯一 write credential を持つ proxy（新規 400–600 行）は、バグ = 全権漏洩のセキュリティ境界。その設計・実装・テスト・レビューが全部同一人物。
- **なぜ**: M13（CI 全量）を義務化しても、**テストを書くのも同じ 1 人**＝自己参照的保証。#279「ザル」は同じ体制・同じ理念（機械的保証）の下で起きた。M11「全部ゲート内」の実効性はゲートの質に依存し、ゲートの質を担保する第二者は存在しない。routines 案でも proxy は自前だが、自作案は加えて dispatcher/spawn/watchdog/heartbeat まで同一 1 人の監査下に置く＝境界面の総量が大きい。

### C-2. 専門面の集中と「動いているが誰も追随していない」腐敗
- **位置**: §4.7 工数の物差し・§6-3
- **何が**: systemd DynamicUser/LoadCredential（§7-6 自認の未検証領域）・GitHub App 管理・Postgres 運用・CC headless 外部仕様追随・Actions cron——各面のドリフト対応がすべて監査役 1 人に集中。
- **なぜ**: §6-3 は一時費用（2.5–3.5k 行）を定量化する一方、**恒久費用（外部仕様追随・E 運用・DB 運用）を「受容の裁定」という定性語に逃がす**——非対称な会計。lathe 開発再開後、監査役の関心が製品に戻った瞬間に最初に腐るのが自前カーネル（現行系の stale 常駐 S2-1 は、まさに「係が不在」で起きた）。routines 案は発火・環境・版の追随を外部組織が恒久に持つ。

## D. 今日の事故クラスのうち自作継続でだけ再発しうるもの

### D-1. E2（OAuth 欠落）類 = ローカル認証への恒久依存
- **位置**: §4.2 runtime・§7-4
- **何が**: ローカル headless は CC のローカル認証に恒久依存。しかも**課金経路（API key か Max サブスク充当か）すら未照合**（§7-4 自認）。token/session 失効は run 途中の silent 失敗として再発しうる（推測だが、クラスは E2 実弾に接地）。
- **なぜ**: cloud 実行なら認証・環境・依存（E1〜E3 全クラス）は platform 管理。両案の §3 集計自身が一致して認める「E 系は routines で消える」群であり、自作継続の固有残余。

### D-2. S1-1（マシン死）一般化と、移行期間の S1-2（二重 dispatch）再発窓
- **位置**: §4.6・§5 Step 1〜6
- **何が**: (a) マシン死クラスは本案自身が「自前 watchdog では原理的に検知不能」と認める固有残余（外部 heartbeat の SLO は未実測）。(b) より鋭い穴: **移行期間（Step 1〜5）は旧系（fs マーカー排他）と新系（DB claim）に共通の排他制約が存在しない**。旧×新の間の二重 dispatch は Step 6（旧系退役）まで構造的に閉じないのに、Step 1 の PoC は**実 issue** を使う。§5 に「PoC 対象 issue を旧系から隔離する手順（gov:hold・旧 timer 停止等）」の明記がない。
- **なぜ**: S1-2 は「guard 追加 eca8247 の**後**に再発」した実績があるクラスで、#294/#295 は Mac×case 併存というまさに「排他制約を共有しない 2 系の並走」で起きた。本案はその再演条件を導入計画自身に内蔵している。

## E. 「作る時間」の機会費用と比較枠の歪み

### E-1. カレンダー見積りの不在（行数は出すが日数を出さない）
- **位置**: §4.7・§5
- **何が**: 2.5–3.5k 行＋テスト 6.2k 行超＋ops/＋contracts 群の構築に、所要**期間**の見積りがない。唯一の先例（bootstrap 15 スライス 4 波 8 PR）は「動く loop の上で inner agent が実装した」実績であり、今回は loop 凍結中＝**監査役の手作業（または壊れた旧系での実装）という鶏卵問題**に触れていない。
- **なぜ**: 実測ベースの推定（推測・幅明記）: 現行系は 32 日で実装 7k＋テスト 6.2k 行。カーネルが半分の規模でも**数週間〜1 か月超の監査役専従**が母数に乗り、その間 lathe 本体は停止したまま。routines 案なら Step 0 spike（1〜2 日）で採否の分水嶺（(d)(g)）が先に判明し、共通自前部品（dedup/watchdog/proxy/envelope）の構築は両案で使い回せる——「先に安い方の不確実性を潰す」順序が機会費用最小。

### E-2. 最強の対抗案（routines hybrid）と一度も比較していない
- **位置**: §1-2（「routines 案の判定 B・D4・OTel 未確認群がこの案では発生しない」）・§3 集計
- **何が**: 本案は routines＝cloud 全面（判定 B・観測劣化）とだけ比較する。しかし routines 案 §2.2 hybrid（統治=platform・実行=ローカル runner）は **local JSONL 100%＝観測無劣化・ingest 変更ゼロ・R5（backend 抽象）維持**を同じく達成する。
- **なぜ**: 本案の 2 大優位のうち**第一（transcript 主権）は hybrid で完全に中和**され、残る差分は「統治面まで自前 vs platform」だけになる。そのとき自作固有に残るのは負担側（M2 最終段・M8・DB 運用・bus factor）が主。第二の優位（M3）も §7-2/§7-6 が自認する通り **Step 0 の existence proof 未取得の条件付き**であり、不成立なら「同一 user＋運用規律」＝routines 案の未解決と同格に落ちる。TL;DR はこれを「構造で建つ」と実測前に販売している（§3 表の [構造]＋[未解決(徹底度)] 併記の方が正直で、TL;DR-3 と整合していない）。比較枠を「自作 vs cloud 全面」に切った時点で結論が半分決まっており、裁定資料としては **「自作 vs hybrid」の行を §3 集計に追加してから** D1 に出すべき。

## 総括（裁定者向け 5 行）

1. 「小さなカーネル」の小ささは会計境界の産物。現行系は同じ人・同じ規律で 32 日 0→7k 行（機械計測）であり、増殖を止める新機構は本案にない。
2. 負担の会計が非対称: 一時費用は定量・恒久費用（E 運用・DB 運用・外部仕様追随・unit 新設）は定性語で §6-3 に逃がしている。
3. bus factor 1 のままセキュリティ境界（proxy）と監視系（watchdog+heartbeat）を全部自前化し、監視系は被監視系と同じ床に立つ（相関故障）。
4. 移行計画自身が S1-2 再発窓（旧 fs 排他×新 DB 排他の非共有、Step 6 まで）を内蔵し、PoC は実 issue を使う。
5. 本案の 2 大優位は、第一が routines hybrid で中和され、第二が Step 0 未実測の条件付き。裁定は「自作 vs hybrid」比較行と Step 0 相当の実測を揃えてから。
